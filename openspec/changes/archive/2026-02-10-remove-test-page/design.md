## Context

`main/main.c` 当前包含约 340 行交互测试页面代码，用于验证 ui_core 框架的手势识别和局部刷新功能。测试已通过验证，现在需要清理测试代码，为生产页面（书架、阅读等）的开发准备干净的入口。

ui_core 框架本身设计为生产级框架，不在本次变更范围内。

## Goals / Non-Goals

**Goals:**
- 从 `main/main.c` 移除全部测试页面相关代码
- 保持 `app_main()` 硬件初始化流程完整
- 保持 ui_core 框架启动流程完整（`ui_core_init()` + `ui_touch_start()` + `ui_core_run()`）
- 清理后代码可编译通过

**Non-Goals:**
- 不修改 ui_core 框架的任何文件
- 不新增任何生产页面
- 不引入条件编译或测试开关机制

## Decisions

### 决策 1：直接删除而非条件编译

**选择**：完全删除测试代码，而非用 `#ifdef TEST_PAGE` 包裹保留。

**理由**：测试页面已完成验证目的，代码通过 git 历史可随时恢复。条件编译会增加维护负担，且测试页面不会在生产构建中使用。

### 决策 2：保留无页面的 ui_core 启动

**选择**：清理后 `app_main()` 仍调用 `ui_core_init()` → `ui_touch_start()` → `ui_core_run()`，但不 push 任何页面。

**理由**：ui_core 的页面栈为空时，事件循环正常运行但不分发事件到页面回调。这是一个有效的空闲状态，后续开发者可直接 `ui_page_push()` 接入生产页面，无需修改初始化流程。

**替代方案**：也可以注释掉 `ui_core_run()`，但这会让下一个变更需要重新启用它，增加不必要的改动。

## Risks / Trade-offs

- **[编译警告]** 移除测试代码后，部分 `#include` 可能变为未使用 → 检查并移除不再需要的头文件（如 `<string.h>`）
- **[空页面栈行为]** ui_core 无页面时的行为需要确认 → 低风险，事件循环只是不调用页面回调
